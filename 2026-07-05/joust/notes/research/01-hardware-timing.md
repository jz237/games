## Joust (Williams Electronics, 1982) — Hardware Timing Research Note\n\n**Purpose:** Ground the physics tick for a faithful browser remake. This note nails the CPU clock, video/interrupt cadence, the game-logic update rate, and the coordinate space — the denominator for all physics constants.\n\n### Sourcing priority\nPrimary/technical sources used, in order of trust: **MAME source** (`src/mame/midway/williams.cpp`, `williams_m.cpp`, `williams_v.cpp`, `williams.h`), then **Sean Riddle's hardware writeup**, then KLOV/PixelatedArcade spec sheets, then forum/blog descriptions. Where they disagree, MAME wins because its timing is derived directly from the emulated CRTC constants.\n\n---\n\n### 1. CPU clock\n- **Motorola MC6809E @ exactly 1.000 MHz.**\n- MAME: `MASTER_CLOCK = XTAL(12'000'000)`, then `MC6809E(config, m_maincpu, MASTER_CLOCK/3/4)` → 12,000,000 / 12 = **1,000,000 Hz**. (Category: measured, high confidence.)\n- Corroborated by Sean Riddle (\"6809E clocked at 1 MHz\") and PixelatedArcade (\"MC6809E @ 1.0 MHz\").\n\n### 2. Video timing (≈60 Hz)\n- **Pixel/dot clock = 8.000 MHz** (`MASTER_CLOCK*2/3`).\n- MAME `set_raw(8MHz, 512 htotal, hvis 6..298, 260 vtotal, vvis 7..247)`.\n- **Refresh = 8,000,000 / (512 × 260) = 60.096154 Hz.** Exactly matches PixelatedArcade's `60.096154 Hz`.\n- Frame period ≈ **16.64 ms**. Sean Riddle independently derives this: a 14-bit video counter (1 MHz) reloads to `0x3F00` on first overflow → 16,640 counts = 16.64 ms.\n- Casual \"60 Hz\" is just rounding; use 60.096 for fidelity.\n\n### 3. Visible resolution vs playfield coordinate space (CONFLICT — resolved)\n- **Visible display: 292 × 240 pixels.** Horizontal visible = 6..298 (292 px), vertical = 7..247 (240 lines). Joust uses the base `williams_base` timing and does **not** override `set_visarea` (call chain: `joust()` → `williams_b1()` → `williams_base()`). Matches KLOV/PixelatedArcade 292×240.\n- **Full addressable video-RAM bitmap: 304 × 256, 4 bits/pixel** (Sean Riddle). Two pixels per byte → 152 bytes × 256 rows ≈ 38K of the 48K DRAM. This is the coordinate space the game code actually writes into; the edges fall outside the visible window (overscan).\n- **Resolution:** treat **292×240 as the visible window** and **~304×256 as the internal playfield coordinate space**. Both numbers are correct; they describe different things. Trust MAME for the visible figure.\n\n### 4. Interrupt / IRQ cadence (the heart of the tick)\nTwo independent scanline timers drive the single 6809 IRQ line (merged via `INPUT_MERGER_ANY_HIGH` → `M6809_IRQ_LINE`):\n\n1. **VA11 scanline IRQ** — `scan_timer` fires every 32 scanlines and sets PIA1 **CB1 = bit 5 of the scanline** (`va11_callback`: `m_pia[1]->cb1_w(BIT(scanline,5))`). CB1 therefore flips state at lines 32, 64, 96, 128, 160, 192, 224, 256 → **8 edges per frame**; the PIA raises an IRQ on these transitions. These are mid-frame housekeeping/raster IRQs.\n2. **COUNT240 IRQ** — `240_timer` fires at scanline 240 and sets PIA1 **CA1** (`count240_callback`: `m_pia[1]->ca1_w(scanline >= 240 ? 1 : 0)`), i.e. the logical AND of VA10–VA13. This is the classic **once-per-frame, end-of-active-video interrupt** — the signal that \"the visible frame is done, run the game update.\"\n\nThe CPU can also **poll the video counter** any time: MAME returns `m_screen->vpos() & 0xFC` (top 6 bits, VA2–VA13).\n\n**CONFLICT (resolved):** blogs/forums describe the scanline IRQ as \"once per quarter of the screen\" (4/frame). That conflates VA11 (8 edges/frame) with COUNT240 (1/frame). Trust MAME: the load-bearing per-frame game tick is **COUNT240**; VA11 edges are extra mid-frame IRQs.\n\n### 5. Game-logic update rate (the physics tick)\n- **The main game loop advances all object physics once per video frame = 60 Hz (60.096 Hz precisely).** Williams 6809 games run a frame-locked simulation tied to the COUNT240 end-of-frame interrupt. (Category: well-known-fact + MAME-inferred; **not** yet verified against an annotated ROM disassembly, so medium confidence on \"exactly every frame with no skips.\")\n- No evidence of a sub-multiple (e.g. 30 Hz) tick for core movement. Open item: confirm via the 6809 disassembly whether heavy frames ever skip an update.\n\n### 6. Supporting hardware (not tick sources)\n- **Sound:** MC6808 on the older board, `SOUND_CLOCK = 3.579545 MHz` XTAL, internal /4 → ~0.895 MHz. Independent of game timing.\n- **Blitter:** two 4-bit \"Special Chip\" blitters (Joust = `WILLIAMS_BLITTER_SC1`) move rectangular blocks (source/dest/width/height) without CPU help. **No hardware sprites** — pure bitmap framebuffer.\n- **Watchdog:** CMOS watchdog resets the board if not serviced within ~133 ms.\n\n---\n\n### For the remake — concrete numbers to implement\n- **Physics tick: 60 Hz** (use **60.096 Hz** if you want exact arcade fidelity; plain 60 is fine). Advance **all** game logic exactly **once per tick**.\n- **Denominator for every physics constant: 1 tick = 1/60.096 s ≈ 16.64 ms.** Express velocities/gravity/acceleration as **pixels-per-tick**.\n- **Coordinate space: internal playfield 304 × 256**, render a **292 × 240** visible window from it.\n- **Decouple from the display:** run a fixed-timestep accumulator so the 60 Hz simulation is independent of the browser's `requestAnimationFrame` rate (handles 120/144 Hz monitors correctly).\n- **Ignore** VA11 mid-frame IRQs and the watchdog for gameplay — they don't affect the once-per-frame simulation model.\n\n### Sources\n- MAME williams driver — https://github.com/mamedev/mame/blob/master/src/mame/midway/williams.cpp (raw: https://gitlab.com/mamedev/mame/-/raw/master/src/mame/midway/williams.cpp)\n- MAME williams machine (interrupt callbacks) — https://gitlab.com/mamedev/mame/-/raw/master/src/mame/midway/williams_m.cpp\n- MAME williams video (counter read) — https://gitlab.com/mamedev/mame/-/raw/master/src/mame/midway/williams_v.cpp\n- Sean Riddle, Williams hardware — https://seanriddle.com/willhard.html ; blitter — https://seanriddle.com/blitter.html\n- PixelatedArcade Joust tech specs — https://pixelatedarcade.com/games/joust/techspecs\n- Glen's Weblog, Williams scanline interrupt — https://nowhereman999.wordpress.com/2020/11/22/williams-hardware-scanline-interrupt-and-a-new-way-to-update-sprites-on-the-coco3/
